# Ревью текущей архитектуры проекта

## 1) Что сейчас есть (карта проекта)

Проект — Next.js 14 (App Router) + Prisma + Zustand + axios.

### Слои, которые уже просматриваются

- **UI-слой**: `src/components/ui/*`, `src/components/shared/*`, `src/components/widgets/*`.
- **App-слой (роуты и API)**: `src/app/*`, `src/app/api/*`.
- **Data access**: Prisma-схема и клиент в `prisma/*`.
- **Клиентские запросы**: `services/*`.
- **Локальный state**: `store/store.ts` + локальные `useState`/хуки.

Это хорошая база: стек подобран адекватно для e-commerce витрины.

---

## 2) Сильные стороны

1. **Современный стек**: Next.js App Router + Prisma + TypeScript.
2. **Разделение UI на уровни** (`ui/shared/widgets`) уже есть и можно развивать.
3. **API-роуты отделены от UI**, есть зачаток `services` как клиента к API.
4. **Prisma-схема предметной области** довольно полная (user/cart/order/product/...)

---

## 3) Главные архитектурные проблемы на текущем этапе

### 3.1. Смешение слоёв и непоследовательная структура

- В корне есть `hooks/`, `services/`, `store/`, а в `src` — основные компоненты и app-роутинг.
- Из-за этого сложно понять «каноничное место» для нового кода.

**Риск:** масштабирование команды и фич становится дорогим.

### 3.2. Непоследовательный нейминг и опечатки

- Пример: `useIngrediets` (опечатка), `category_state/category_change` в snake_case в TS-коде.
- Микс `Container.tsx` и импортов через `./container` может ломаться на case-sensitive FS.

**Риск:** скрытые баги на CI/production и падение читабельности.

### 3.3. Слабая доменная модель на фронте

- В `PizzaList` данные захардкожены.
- Фильтры пока почти не связаны с URL/API и доменными сущностями (`Product`, `ProductItem`, `Ingredient`).

**Риск:** при росте фич появится дублирование и сложная синхронизация состояния.

### 3.4. Сервисный слой слишком тонкий

- `services` сейчас фактически wrapper над axios без валидации/нормализации/обработки ошибок.
- Нет единого контракта ошибок и typed mapping между API DTO и UI model.

**Риск:** ошибки сети и изменения API «протекают» в компоненты.

### 3.5. Отсутствует единый подход к state management

- Где-то Zustand, где-то локальный стейт, где-то custom hook с side effect.
- Нет правил: что хранить в URL, что в server state, что в UI state.

**Риск:** несогласованное поведение фильтров и сложная отладка.

### 3.6. Недостаток архитектурной документации

- README типовой, не объясняет структуру и правила развития.

**Риск:** высокий onboarding cost.

## 4) Рекомендуемая целевая архитектура (эволюционно, без «биг-банга»)

Ниже вариант, который хорошо ложится на ваш текущий код.

```text
src/
  app/                  # маршруты Next.js, layout/page/route
  shared/               # общие утилиты, ui-kit, lib, api instance
  entities/             # доменные сущности (product, ingredient, user)
    product/
      model/
      api/
      ui/
  features/             # пользовательские фичи (filter-products, sort-products)
    filter-products/
      model/
      ui/
      lib/
  widgets/              # композиция фич и entities в крупные блоки страницы
  processes/            # (опционально) длинные пользовательские процессы
```

### Принципы

1. **Feature-first**: код группируется по фичам/сущностям, а не по типу файла.
2. **Явные границы**: `app` не знает внутренности data-layer, только публичные API модулей.
3. **Typed contracts**: DTO/Schema (zod) на входе/выходе API.
4. **URL как источник правды для фильтров каталога** (query params).

## 5) Практический план улучшений

### Этап 1 (быстрые победы, 1–2 дня)

1. Выбрать единый нейминг: `camelCase` для функций/полей (`categoryState`, `setCategory`).
2. Исправить опечатки (`useIngredients`) и нормализовать имена компонентов.
3. Привести структуру к одному корню (`src/*`), перенести `hooks/services/store` в `src/shared` или `src/features`.
4. Обновить README: как устроен проект и куда добавлять новый код.

### Этап 2 (каталог и фильтры, 2–4 дня)

1. Вынести модель фильтров в отдельный модуль `features/filter-products/model`.
2. Синхронизировать фильтры с query params (`useSearchParams`, `router.replace`).
3. Подключить реальный список продуктов из API вместо хардкода.
4. Добавить базовую обработку loading/error/empty-состояний.

### Этап 3 (устойчивость, 3–5 дней)

1. Ввести runtime-валидацию ответов API (zod).
2. Добавить централизованный error mapping в `services`.
3. Ввести минимальные архитектурные правила (линт-правила импортов / boundaries).
4. Написать smoke-тесты для API-роутов и критичных UI-потоков.

---

## 6) Что у вас уже можно использовать как «опорные точки»

- Prisma-схема уже отражает основной домен и может быть источником типов и контрактов.
- Разделение `shared/ui` и `widgets` уже есть — можно постепенно довести до feature-модульности.
- API-роуты уже заведены, значит переход от моков к реальным данным не потребует смены платформы.

---

## 7) Краткий итог

Сейчас проект в рабочем MVP-состоянии: хороший стек и зачатки слоистой структуры есть, но пока много технической неоднородности (нейминг, структура каталогов, state-flow, контракты API). Лучший путь — **не переписывать всё**, а сделать 3 коротких этапа: сначала консистентность и структура, потом фильтры/каталог, затем устойчивость и тесты.

## 8) Критичные блокеры, найденные при проверке сборки

1. **Ошибки импортов из-за регистра имени файла**: используется путь `container`, а файл называется `Container.tsx`.
2. **Сборка зависит от внешнего Google Fonts** (`Nunito`) и падает при недоступности `fonts.googleapis.com`.

Рекомендация: сначала устранить эти блокеры (импорты/шрифты), затем двигаться по этапам из раздела 5.
